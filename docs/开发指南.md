项目开发指南 — 给成员的快速上手说明
======

**目的**: 帮助组员快速理解项目组成、当前已提供的代码/文档、各自需要完成的任务，以及如何运行/编写 `tests/` 下的单元测试。

**项目总体结构**
- **Engine (游戏引擎)**: 状态管理、回合流程、卡牌/技能效果。
  - 主要文件：[include/game_engine.h](include/game_engine.h), [src/game_engine.c](src/game_engine.c), [src/cards.c](src/cards.c)
- **Skills / Cards (技能与卡牌实现)**: 武将技能与卡牌效果实现。
  - 主要文件：[src/skills.c](src/skills.c), [src/cards.c](src/cards.c)
- **AI (决策模块)**: 人格驱动的决策与心理文字生成。
  - 主要文件：[include/ai.h](include/ai.h), [src/ai_decision.c](src/ai_decision.c), [src/ai_psych.c](src/ai_psych.c)
- **UI (用户界面)**: 控制台或 EasyX 窗口展示与玩家输入。
  - 主要文件：[include/ui.h](include/ui.h), [src/ui.c](src/ui.c)
- **File IO (数据读写/统计)**: 加载配置（cards.txt、personalities.txt）、保存对局结果。
  - 主要文件：[include/file_io.h](include/file_io.h), [src/file_io.c](src/file_io.c)
- **Tests (测试)**: 每个模块对应 `tests/test_*.c` 的单元/集成测试。
  - 目录：[tests/](tests/)

**我已提供的内容（当前仓库里可直接使用）**
- 接口与头文件：`include/` 下定义了 `GameState`、卡牌/武将/人格枚举和 Engine/AI/UI/FileIO 的函数声明（见 [include/game_engine.h](include/game_engine.h) 等）。
- 引擎骨架：`src/game_engine.c` 中已填入基础实现（初始化、回合推进、卡牌效果调用桩）。
- 技能桩实现：`src/skills.c` 提供了武将技能的最小实现以便测试编译通过。
- AI 与 FileIO 最简实现：`src/ai_decision.c`，`src/file_io.c` 含基本可用的桩/文件读写示例。
- 控制台 UI：`src/ui.c` 提供了控制台版绘制与输入，便于本地调试。
- 测试骨架：`tests/` 下包含 `test_cards.c`、`test_skills.c`、`test_ai.c`、`test_ui.c`、`test_file_io.c`、`test_engine.c` 及 `tests/test_utils.h`。这些测试目前是可编译的示例，但需要你们完善断言与用例。
- Makefile：已添加 `test` 目标，可通过 `make test` 或直接用 `gcc` 编译 `tests/*.c` 生成可执行测试二进制（Windows 上可能使用 MinGW 下的 `mingw32-make` 或直接用 `gcc` 命令）。

**每个模块的开发任务与优先级（建议分工）**
- **Engine（优先级：高）**
  - 完成 `game_init` 的模式细化（单挑 / 主公局）、角色阵营与主公逻辑。
  - 实现回合流程（准备→摸牌→出牌→弃牌→结束）与濒死求桃流程。
  - 实现 `game_get_legal_actions`、`game_perform_action` 的完整判定与胜负判定。
  - 负责与 AI、UI、FileIO 的接口稳定性。

- **Cards / Skills（优先级：中）**
  - 在 `src/cards.c` / `src/skills.c` 中实现卡牌效果与武将技能（张飞、赵云、诸葛亮）。
  - 保持函数签名与 `include/game_engine.h` 一致，编写局部单元测试覆盖常见情况（如杀/闪交互、桃的濒死救人）。

- **AI（优先级：中）**
  - 在 `src/ai_decision.c` 中实现 `ai_decide_action`，依据 `Personality` 调整评分函数。
  - 在 `src/ai_psych.c` 中实现心理文字模板，提供 `ai_get_last_psych_message()` 与 `ai_get_last_emotion()`。
  - 编写 `tests/test_ai.c` 中的案例，验证不同人格下的决策倾向与心理文案返回。

- **UI（优先级：低→中）**
  - 控制台 UI（`src/ui.c`）先保持并完善交互细节；若时间充足，移植到 EasyX 图形界面。
  - 确保 `ui_get_player_action` 返回的 `Action` 与 Engine 接口兼容。

- **File IO（优先级：低）**
  - 补全 `load_card_deck`、`load_heroes`、`load_personalities` 以支持自定义数据文件 `data/cards.txt` 与 `data/personalities.txt`。
  - 实现 `save_game_result` 将对局结果记录到 `game_results.txt`，供后续统计使用。

**如何使用 tests（快速上手）**
- 目录：打开仓库的 `tests/`，每个测试对应一个 `.c` 文件，习惯上 `tests/test_*.c` 测试单个模块，`tests/test_engine.c` 为集成测试。
- 推荐运行方式（Windows + MinGW 下）:
```powershell
cd c:\Users\15352\Desktop\Sanguosha-GameAI
gcc -Wall -g tests/test_engine.c src/ai_decision.c src/ai_psych.c src/cards.c src/file_io.c src/game_engine.c src/skills.c src/stats.c src/ui.c -Iinclude -o tests/test_engine.exe
./tests/test_engine.exe
```
或使用 Makefile（在支持 make 的环境下）:
```bash
make test
```
- 如果链接到 `main` 冲突（测试文件也包含 `main()`），请确保构建时不包含 `src/main.c`。
- 编写/扩展测试时请遵守：
  - 测试应独立（构造确定性 `GameState`），避免依赖随机数。
  - 对缺失依赖使用桩（stub）函数局部实现，放在测试文件顶端或 `tests/stubs.c`。
  - 每个测试用例用 `TEST_START` / `TEST_PASS` 包裹，并用 `ASSERT_*` 明确断言失败信息。

**提交流程与代码审查（轻量）**
- 分支策略：每个功能开一个分支 `feature/<模块>-<简述>`，如 `feature/engine-turnflow`。
- 提交信息模板：`<模块>: <简短说明>`，例如 `engine: 完成游戏回合推进逻辑`。
- 发起 Pull Request 后：
  - 运行 `make test`，在 PR 描述贴上测试通过截图或日志摘要。
  - 至少一位组员（非作者）复查代码风格与接口变更。
