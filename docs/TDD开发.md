项目测试方案设计文档（讨论稿）
一、背景与目标
本小组正在开发“三国杀·人格分裂AI”项目，采用多文件模块化结构。组员开发经验有限，传统依赖“写完再集成调试”的方式容易导致问题难以定位。为了降低协作成本、提高开发信心，我们提出为每个核心 .c 文件建立独立的单元测试，放在 tests/ 目录下。每个开发者可以随时运行自己的测试，快速验证所写功能是否符合预期。

核心目标：

每个模块（cards.c、skills.c、ai.c、ui.c、file_io.c）都有对应的测试文件。

对于两人协作的 Engine 部分，除了整体的 test_engine.c，还要为 cards.c 和 skills.c 提供独立的测试，避免互相阻塞。

测试通过即视为该模块功能完成，后续集成时只要所有模块测试通过，整体大概率能正常工作。

二、整体测试层次
测试层次	对应文件	测试对象	依赖关系
单元测试	test_cards.c	cards.c 中的卡牌效果函数	需要 GameState 结构体及少量桩函数（如 draw_card, save_dying）
单元测试	test_skills.c	skills.c 中的武将技能函数	需要 GameState 及可能涉及 game_get_legal_actions 的桩
单元测试	test_ai.c	ai.c 中的决策与心理文字函数	需要 game_get_legal_actions 的桩，初期可以不依赖真实 Engine
单元测试	test_ui.c	ui.c 的绘图与输入函数	无外部依赖（控制台版）；可手动或自动模拟输入
单元测试	test_file_io.c	file_io.c 的所有文件读写、统计函数	无其他模块依赖，只依赖标准库
集成测试	test_engine.c	game_engine.c + cards.c + skills.c 联合	依赖完整的 Engine 实现，测试整体回合、濒死、胜负等
说明：test_engine.c 可以视为对 B 同学（主写 Engine）和 D 同学（协助 cards/skills）的联合验收。但在此之前，两人可以通过各自的单元测试独立工作。

三、每个测试文件的设计原则
1. 独立性
每个测试文件应能单独编译链接，不依赖尚未完成的模块。

对于缺失的外部函数，使用桩函数（stub）提供简单实现（例如返回固定值、打印日志）。

2. 可重复性
测试中使用的初始游戏状态应是确定的（手动构造，不依赖随机数）。

每个测试用例执行后，应恢复环境或重新构造状态，避免相互影响。

3. 清晰的断言与错误报告
使用 assert 宏或自定义断言函数，在失败时打印文件、行号、期望值和实际值。

例如：ASSERT_EQ(actual_hp, expected_hp, "杀效果后目标血量错误");

4. 桩函数策略
在测试文件中直接定义桩函数（使用 static 避免符号冲突），或单独提供 stubs.c 供多个测试复用。

桩函数只需满足被测试模块的调用约定，返回合理的默认值（例如 draw_card 可以什么都不做，save_dying 返回 0 表示不救）。

四、各测试文件具体职责
测试文件	主要测试内容	通过标准示例
test_cards.c	apply_card_effect 对杀、桃、过拆、无中的效果	- 杀命中后目标血量减1
- 桃使使用者回血且不超过上限
- 过拆使目标手牌数减1
- 无中使使用者手牌数+2
test_skills.c	张飞无限杀、赵云转化、诸葛亮观星（简化版）	- 张飞：第一张杀后，第二张杀仍合法
- 赵云：杀闪互转（若实现）
- 诸葛亮：摸牌阶段多摸一张
test_ai.c	ai_decide_action 在不同人格下的倾向，ai_get_last_psych_message 内容	- 激进派优先出杀
- 保守派优先出桃或闪
- 赌徒派行动带有随机性但仍合理
- 心理文字包含关键信息
test_ui.c	ui_draw_game 输出内容完整，ui_get_player_action 正确解析输入	- 打印包含双方血量、手牌列表
- 输入 1+卡牌编号+目标 返回正确 Action
- 输入 2 返回结束回合
test_file_io.c	配置文件加载、对局保存、胜率统计	- 读取 cards.txt 得到正确牌堆
- 保存结果后文件追加一行
- 统计胜率数值正确
test_engine.c	完整回合、濒死、胜负、主公模式	- 一回合内出杀-闪逻辑正确
- 濒死求桃顺序正确
- 主公死亡反贼胜利

五、测试目录与工作方式

为了让每位组员能够清楚自己的开发方向，建议把测试文件统一放在 `tests/` 目录下，每个模块对应一个测试文件：

- `tests/test_cards.c`
- `tests/test_skills.c`
- `tests/test_ai.c`
- `tests/test_ui.c`
- `tests/test_file_io.c`
- `tests/test_engine.c`

如果已有测试文件（如根目录下的 `test_ui.c`），可以先迁移到该目录，再作为统一测试体系的一部分。

每个测试文件应包含：

- 本模块的功能目标描述
- 具体的测试用例
- 失败时清晰的错误输出
- 对未完成依赖的桩函数支持



六、测试如何帮助组员

测试能帮助每位组员：

- 快速确认自己实现的接口是否符合预期
- 释放“我先完成整个系统再集成”的压力
- 在本模块范围内反复验证，减少对其他模块的依赖等待
- 通过失败提示直接定位问题，而不是盲目调试
