你的模块划分非常合理，而且加上了“表情交互”这种特色，会让作品更有记忆点。下面我按照你的想法，给出**完整、可并行开发**的模块设计，包括每个模块的职责、对外接口、数据流向，以及开发时的依赖关系。

## 省流总结

- 核心拆成 4 个模块：UI、Engine、AI、文件操作。
- UI 负责画面与输入，Engine 负责状态与规则，AI 负责智能决策，文件操作负责配置与统计。
- 模块间只约定接口，不绑定实现，便于并行开发与后期扩展。
- 先搭好头文件与引擎骨架，再补AI逻辑、UI展示与数据读写。

## 角色分工

- A同学：前端界面（UI），包括EasyX绘图、输入处理、心理活动与表情显示。
- B同学和D同学共同负责：
  - 游戏引擎（Engine），负责游戏状态管理、回合流程、卡牌效果和胜负判定。
  - 文件操作（Data），负责配置加载、结果保存、统计管理和日志存储。
- C同学：AI玩家（AI），负责人格权重、行动评估、目标选择和心理文字生成。

## 二、模块详细职责与接口

### 模块1：前端界面（UI）

| 文件 | 职责 |
| --- | --- |
| `ui.c` / `ui.h` | EasyX窗口管理、绘图、鼠标点击检测、按钮状态 |
| `表情包资源` | 若干张表情图片（PNG），对应不同情绪（笑、怒、奸笑、冷汗等） |

**对外接口（供主函数调用）**：

```c
// 初始化窗口，加载资源
void ui_init();
// 绘制主游戏界面（根据GameState绘制）
void ui_draw_game(GameState *gs);
// 等待玩家操作，返回玩家选择的行动（出哪张牌、对谁、结束回合等）
PlayerAction ui_get_player_action(GameState *gs);
// 显示心理活动文字 + 对应表情（AI调用时传入）
void ui_show_psych_message(char *message, int emotion_id);
// 显示游戏结束画面
void ui_show_gameover(int winner);
// 关闭窗口
void ui_close();
```

**表情映射建议**：

- 激进派出杀 → 愤怒/战意表情
- 保守派吃桃 → 得意/眨眼
- 赌徒派出牌 → 奸笑/赌徒眼神
- AI濒死 → 惊恐表情

> 表情可以用EasyX的 `loadimage` 加载小图片，在心理活动文字旁边绘制。

### 模块2：游戏引擎（Engine）

| 文件 | 职责 |
| --- | --- |
| `game.h` | 游戏状态结构体（GameState）、角色结构体、卡牌结构体等 |
| `game_engine.c` | 回合控制、摸牌、出牌合法性、效果执行、濒死求桃 |
| `skills.c` | 武将技能实现（张飞、赵云、诸葛亮） |
| `cards.c` | 卡牌效果（杀伤害、过拆、无中、桃） |

**对外接口（供UI和主循环调用）**：

```c
// 初始化新游戏（模式、武将选择、人格随机）
void game_init(GameState *gs, int mode, int player_hero, int ai_hero, Personality ai_psn);
// 执行一个行动（由玩家或AI发起），返回结果（是否合法，是否游戏结束）
ActionResult game_perform_action(GameState *gs, Action action);
// 检查当前回合是否应该自动结束（如无牌可出或玩家点了结束）
int game_is_turn_over(GameState *gs);
// 强制结束当前回合，切换到下一位
void game_next_turn(GameState *gs);
// 获取当前合法行动列表（用于AI决策时遍历）
ActionList game_get_legal_actions(GameState *gs, Character *actor);
```

**内部核心函数（供上述接口调用）**：

```c
// 摸牌
void draw_card(GameState *gs, Character *ch, int num);
// 使用卡牌的效果（不检查合法性，仅执行）
void apply_card_effect(GameState *gs, Character *user, Card card, Character *target);
// 濒死求桃
int save_dying(GameState *gs, Character *dying, int start_index);
```

### 模块3：AI玩家模块

| 文件 | 职责 |
| --- | --- |
| `ai.h` | 人格枚举、权重结构体 |
| `ai_decision.c` | 评分函数、最佳行动选择、目标选择 |
| `ai_psych.c` | 动态心理文字生成、表情选择 |

**对外接口（供游戏引擎在AI回合调用）**：

```c
// AI决策：返回一个行动（出哪张牌、目标是谁）
Action ai_decide_action(GameState *gs, Character *ai);
// 在濒死求桃时，AI决定是否使用桃救某个角色
int ai_should_use_tao_to_save(GameState *gs, Character *saver, Character *dying);
// 获取AI的心理活动字符串（在ai_decide_action内部生成，外部通过getter获取）
char* ai_get_last_psych_message();
int ai_get_last_emotion();  // 返回表情ID
```

**内部实现**：

- 加载人格权重表（从文件或硬编码）。
- 对每个合法行动调用 `evaluate_action` 计算分数，选择最高分。
- 生成心理文字时，根据人格、所出牌型、双方状态拼接动态句子，并决定表情ID。

### 模块4：文件操作模块（数据持久化）

| 文件 | 职责 |
| --- | --- |
| `file_io.h` | 定义配置文件路径、数据格式 |
| `file_io.c` | 读/写武将、人格参数、牌堆配置 |
| `stats.c` | 统计管理（保存对局结果、查询胜率、排序） |

**对外接口（供主程序、游戏引擎、后台管理调用）**：

```c
// 加载武将配置（heroes.txt），返回武将列表
int load_heroes(Hero heroes[], int max_count);
// 加载人格参数（personality.txt）
int load_personalities(PersonalityWeights weights[], int max_count);
// 加载牌堆配置（cards.txt）
int load_card_deck(CardDeck *deck);
// 保存一局游戏结果（games.log）
void save_game_result(char *mode, char *player_hero, char *ai_hero, char *ai_psn, int winner);
// 统计功能：读取games.log，返回胜率等
Stats get_stats();
// 后台管理：保存修改后的武将/人格参数（覆盖文件）
int save_heroes(Hero heroes[], int count);
int save_personalities(PersonalityWeights weights[], int count);
```

> 后台管理菜单可单独做一个控制台程序或集成在主菜单中，调用这些接口。

## 三、模块间依赖关系与开发顺序

### 依赖图（箭头表示依赖）

```
UI  →  Engine (调用游戏API)
AI  →  Engine (需要获取合法行动、评估状态)
Engine → AI (在AI回合调用AI决策)
Engine → FileIO (初始化时加载配置，结束时保存)
UI  →  FileIO (后台管理菜单)
```

### 建议开发顺序（并行友好）

| 阶段 | 任务 | 依赖 | 可并行 |
| --- | --- | --- | --- |
| 1 | 定义所有头文件（数据结构、接口） | 无 | 全员一起 |
| 2 | 实现游戏引擎核心（不依赖AI和UI，用固定输入测试） | 无 | B同学 |
| 3 | 实现文件操作模块（读写配置） | 无 | D同学 |
| 4 | 实现AI决策（先不做心理活动，只做分数选择） | Engine头文件 | C同学 |
| 5 | 实现UI（控制台版本，便于测试；EasyX后期包装） | Engine头文件 | A同学 |
| 6 | 集成调试（单挑模式，AI为固定人格） | 2,4,5 | 全体 |
| 7 | 加入心理活动和表情 | AI模块 | C同学，A同学配合 |
| 8 | 加入非对称模式、主公增益 | Engine | B同学 |
| 9 | 后台管理菜单 | UI, FileIO | A, D |
| 10 | EasyX图形美化 | UI | A |

## 四、关键数据结构定义（示例，供头文件使用）

```c
// game.h
typedef struct {
    int type; // CARD_SHA, ...
    int id;
} Card;

typedef struct {
    HeroType hero;
    Personality personality; // 仅AI有效
    int hp, max_hp;
    Card hand[10];
    int hand_count;
    int is_ai;
    int is_lord;
} Character;

typedef struct {
    Character player;
    Character ai[2];
    int ai_count;
    int current_turn; // 0玩家,1 AI0,2 AI1
    Card draw_pile[50];
    int pile_count;
    Card discard_pile[50];
    int discard_count;
    int mode; // 0单挑,1玩家主公,2AI主公
    int game_over;
    int winner;
} GameState;

typedef struct {
    int action_type; // ACTION_PLAY, ACTION_END_TURN
    int card_index;  // 手牌索引
    Character *target; // 目标
} Action;
```

## 五、如何体现“表情交互”

- 准备 6~8 个小图片（例如：怒、笑、惊、汗、阴险、呐喊），每个 20x20 像素。
- 在UI的心理活动区域旁边，根据AI返回的 `emotion_id` 显示对应图片。
- 可以配合文字气泡效果（画圆角矩形）。

> 表情不需要太复杂，甚至可以用 Emoji 字符（😠😊😨）代替图片，先保证功能。

## 六、总结

这个模块划分：

- **职责清晰**：每个人只关心自己的模块，通过接口交互。
    
- **并行开发**：Engine、AI、FileIO可以同时进行，UI前期用控制台模拟，后期替换EasyX。
    
- **符合课程要求**：文件操作、智能策略、界面（含制作者信息）都明确分配。
    
- **特色突出**：人格分裂 + 心理活动交互 = 答辩神器。
